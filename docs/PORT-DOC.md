> **Версия**: 2.0  
> **Исходный проект**: Next.js 14 + TypeScript + TanStack Query Web Client  
> **Создано**: 2026-03-15  
> **Обновлено**: 2026-03-15  
> **Назначение**: Спецификация для реконструкции приложения управления задачами GTD на любой целевой платформе.

---

## 1. Обзор проекта

### Что делает этот проект
Веб-приложение для управления личными задачами, реализующее методологию Getting Things Done (GTD). Пользователи могут захватывать задачи в Inbox, уточнять и организовывать их по проектам, контекстам, тегам и областям, фокусироваться на следующих действиях и проводить еженедельные обзоры.

### Основные персоны пользователей
- **Фокусированный на продуктивности индивидуум**: Нужен систематический подход к управлению личными задачами в разных контекстах и проектах
- **Практик GTD**: Знаток методологии GTD, нуждающийся в инструментах, поддерживающих пятишаговый рабочий процесс (Capture, Clarify, Organize, Engage, Review)

### Ключевые ограничения и обязательные требования
- Требуется внешний сервер Todo API для работы (постоянство данных обрабатывается на стороне сервера)
- Должна поддерживаться оптимистичная реакция в реальном времени для операций с задачами
- Должна поддерживаться полная реализация рабочего процесса GTD
- Состояние приложения является временным на стороне клиента; постоянные данные хранятся в API

## 2. Инвентарь функций

### Функция: Быстрое создание задачи
- **ID**: F-001
- **Краткое описание**: Быстрое создание задачи с минимальным вводом
- **Пользовательская история**: Как пользователь, я хочу быстро захватывать задачи, чтобы не забывать их
- **Входы**: Заголовок задачи (обязательно), необязательное описание
- **Выходы / Эффекты**: Создаёт задачу со статусом "inbox" в системе
- **Бизнес-правила**:
  - Заголовок обязателен (1-200 символов)
  - Статус по умолчанию: "inbox"
  - Приоритет по умолчанию: "medium"
- **Случаи ошибок**: Ошибка соединения с API, ошибки валидации
- **Внешняя зависимость**: Todo API (POST /inbox)

### Функция: Обработка Inbox
- **ID**: F-002
- **Краткое описание**: Обработка элементов Inbox по одному или всех сразу
- **Пользовательская история**: Как пользователь, я хочу уточнять и организовывать элементы Inbox, чтобы мои задачи были правильно структурированы
- **Входы**: Действия пользователя над каждым элементом (уточнить, удалить, переместить в someday, установить waiting)
- **Выходы / Эффекты**: Задачи перемещаются из Inbox в организованные состояния
- **Бизнес-правила**:
  - Можно обрабатывать одну задачу за раз или все последовательно
  - Уточнение задачи открывает форму для установки проекта, контекста, тегов, приоритета
  - Удаление удаляет задачу из Inbox
  - Режим Process All автоматически переходит к следующей задаче после сохранения
- **Случаи ошибок**: Ошибка API должна показывать ошибку и разрешать повтор
- **Внешняя зависимость**: Todo API (GET /inbox, PUT /tasks/{id}, DELETE /tasks/{id})

### Функция: Список задач с фильтрами
- **ID**: F-003
- **Краткое описание**: Просмотр и фильтрация задач по различным критериям
- **Пользовательская история**: Как пользователь, я хочу фильтровать задачи, чтобы фокусироваться на определённых подмножествах
- **Входы**: Выборы фильтров (status, priority, project, context, tag, текст поиска), опции сортировки
- **Выходы / Эффекты**: Отображает отфильтрованный список задач с пагинацией
- **Бизнес-правила**:
  - Несколько фильтров могут применяться одновременно
  - Поиск совпадает с заголовками задач
  - Опции сортировки: created_at, priority, title (по возрастанию/убыванию)
  - Пагинация: 10-50 элементов на странице
- **Случаи ошибок**: Недопустимые значения фильтров отклоняются
- **Внешняя зависимость**: Todo API (GET /tasks с параметрами запроса)

### Функция: Действия с задачами
- **ID**: F-004
- **Краткое описание**: Выполнение действий над задачами (завершить, редактировать, удалить, установить следующее действие, установить ожидание)
- **Пользовательская история**: Как пользователь, я хочу управлять отдельными задачами, чтобы продвигаться в работе
- **Входы**: Выбор пользователем действия над задачей
- **Выходы / Эффекты**: Состояние задачи изменяется соответственно
- **Бизнес-правила**:
  - Завершить: Устанавливает статус "completed", удаляет из следующих действий
  - Установить следующее действие: Переключает флаг is_next_action. Если задача в waiting/someday/completed, требует подтверждения для изменения статуса на active
  - Установить ожидание: Открывает модальное окно для указания кого/чего мы ожидаем, устанавливает статус "waiting"
  - Редактировать: Открывает форму для изменения всех полей задачи
  - Удалить: Перемещает задачу в корзину (мягкое удаление)
- **Случаи ошибок**: Подтверждения для разрушительных действий, обработка ошибок API
- **Внешняя зависимость**: Todo API (различные конечные точки)

### Функция: Представление следующих действий
- **ID**: F-005
- **Краткое описание**: Фокус на задачах, отмеченных как следующие действия
- **Пользовательская история**: Как пользователь, я хочу видеть свои следующие действия, чтобы знать, что делать сейчас
- **Входы**: Нет (автоматический фильтр для is_next_action=true)
- **Выходы / Эффекты**: Отображает список задач следующих действий
- **Бизнес-правила**:
  - Отображает только задачи с is_next_action=true
  - Завершённые задачи автоматически удаляются
  - Задачи можно переключать как следующие действия из этого представления
- **Случаи ошибок**: Ошибка API
- **Внешняя зависимость**: Todo API (GET /tasks/next-actions)

### Функция: Управление проектами
- **ID**: F-006
- **Краткое описание**: Создание, редактирование, удаление и просмотр проектов
- **Пользовательская история**: Как пользователь, я хочу организовывать задачи по проектам, чтобы отслеживать прогресс
- **Входы**: Название проекта, описание, необязательная область
- **Выходы / Эффекты**: Сущности проектов создаются с автоматическим отслеживанием прогресса
- **Бизнес-правила**:
  - Название проекта обязательно (1-100 символов)
  - Прогресс вычисляется автоматически: (completed_tasks / total_tasks) * 100
  - Проекты могут быть активными или завершёнными
  - Удаление проекта не удаляет связанные задачи
- **Случаи ошибок**: Валидация названия, ошибка API
- **Внешняя зависимость**: Todo API (CRUD на /projects)

### Функция: Представление деталей проекта
- **ID**: F-007
- **Краткое описание**: Просмотр деталей проекта и связанных задач
- **Пользовательская история**: Как пользователь, я хочу видеть все задачи в проекте, чтобы понимать прогресс проекта
- **Входы**: Выбор проекта
- **Выходы / Эффекты**: Показывает информацию о проекте, индикатор прогресса и список задач
- **Бизнес-правила**:
  - Показывает все задачи, принадлежащие проекту
  - Индикатор прогресса отражает процент завершения
  - Можно добавлять новые задачи напрямую в проект
- **Случаи ошибок**: Проект не найден
- **Внешняя зависимость**: Todo API (GET /projects/{id}, GET /tasks?project_id={id})

- **Входы**: Название контекста, необязательное описание

- **Бизнес-правила**:

  - Задачи могут принадлежать максимум одному контексту
- **Случаи ошибок**: Валидация названия
- **Внешняя зависимость**: Todo API (CRUD на /contexts)

### Функция: Управление тегами

- **Выходы / Эффекты**: Сущности тегов доступны для назначения задачам
- **Бизнес-правила**:
  - Название тега обязательно (1-100 символов)

- **Случаи ошибок**: Валидация названия, валидация формата цвета
- **Внешняя зависимость**: Todo API (CRUD на /tags)

### Функция: Управление областями
- **ID**: F-010
- **Краткое описание**: Управление областями ответственности
- **Пользовательская история**: Как пользователь, я хочу определять области, чтобы категоризировать проекты на высоком уровне
- **Входы**: Название области, необязательное описание
- **Выходы / Эффекты**: Сущности областей доступны для назначения проектам
- **Бизнес-правила**:
  - Название области обязательно (1-100 символов)
  - Проекты могут принадлежать максимум одной области
  - Примеры: Карьера, Здоровье, Финансы, Семья
- **Случаи ошибок**: Валидация названия
- **Внешняя зависимость**: Todo API (CRUD на /areas)

### Функция: Еженедельный обзор
- **ID**: F-011
- **Краткое описание**: Пошаговый рабочий процесс еженедельного обзора
- **Пользовательская история**: Как пользователь, я хочу проводить еженедельный обзор, чтобы оставаться организованным
- **Входы**: Навигация пользователя через четыре шага обзора
- **Выходы / Эффекты**: Пользователь обрабатывает Inbox, просматривает проекты, выбирает следующие действия, просматривает элементы Someday
- **Бизнес-правила**:
  - Четырёхшаговый процесс: Обзор Inbox, Обзор проектов, Выбор следующих действий, Обзор Someday
  - Каждый шаг независим, но должен выполняться по порядку
  - Пользователь может свободно перемещаться между шагами
- **Случаи ошибок**: Ошибка API во время обзора
- **Внешняя зависимость**: Todo API (различные конечные точки)

### Функция: Управление корзиной
- **ID**: F-012
- **Краткое описание**: Просмотр, восстановление или окончательное удаление мягко удалённых задач
- **Пользовательская история**: Как пользователь, я хочу восстанавливать удалённые задачи, чтобы не терять данные случайно
- **Входы**: Выбор пользователем действия восстановления или окончательного удаления
- **Выходы / Эффекты**: Задачи восстанавливаются или окончательно удаляются
- **Бизнес-правила**:
  - Мягко удалённые задачи хранятся с временной меткой deleted_at
  - Восстановление очищает временную метку deleted_at
  - Окончательное удаление полностью удаляет задачу
  - Можно опустошить всю корзину сразу
- **Случаи ошибок**: Ошибка API
- **Внешняя зависимость**: Todo API (GET /tasks/deleted, POST /tasks/{id}/restore, DELETE /tasks/{id}/permanent)

### Функция: Обзор дашборда
- **ID**: F-013
- **Краткое описание**: Быстрый обзор ключевых метрик и последних элементов
- **Пользовательская история**: Как пользователь, я хочу видеть свой статус одним взглядом, чтобы планировать день
- **Входы**: Нет (автоматический сбор данных)
- **Выходы / Эффекты**: Отображает счётчики (Inbox, Следующие действия, Активные проекты) и списки (Топ-5 следующих действий, Активные проекты)
- **Бизнес-правила**:
  - Счётчик Inbox: незавершённые задачи со статусом "inbox"
  - Счётчик следующих действий: задачи с is_next_action=true
  - Счётчик активных проектов: проекты со статусом "active"
  - Показывает топ-5 следующих действий (самые последние или с высшим приоритетом)
- **Случаи ошибок**: Ошибка API показывает пустые состояния
- **Внешняя зависимость**: Todo API (несколько конечных точек)

## 3. Модели данных

### Сущность: Задача (Task)

| Поле | Тип | Обязательно | Описание | Ограничения |
|-------|------|------------|-------------|-------------|
| id | integer | Нет (авто) | Уникальный идентификатор | Автогенерируемый |
| title | string | Да | Заголовок задачи | 1-200 символов |
| description | string | Нет | Детали задачи | Максимум 1000 символов |
| completed | boolean | Нет (авто) | Статус завершения | По умолчанию false |
| status | enum | Нет | Статус GTD | inbox, active, completed, waiting, someday (по умолчанию: inbox) |
| priority | enum | Нет | Уровень срочности | low, medium, high (по умолчанию: medium) |
| due_date | datetime | Нет | Срок даты/времени | Формат ISO 8601 |
| reminder_time | datetime | Нет | Время напоминания | Формат ISO 8601 |
| is_next_action | boolean | Нет (авто) | Отмечено как следующее действие | По умолчанию false |
| waiting_for | string | Нет | Кого/чего мы ожидаем | Максимум 200 символов |
| delegated_to | string | Нет | Кому делегирована задача | Максимум 100 символов |
| someday | boolean | Нет | Флаг для someday/maybe | По умолчанию false |
| project_id | integer | Нет | Связанный проект | Должен существовать в таблице projects |
| context_id | integer | Нет | Связанный контекст | Должен существовать в таблице contexts |
| area_id | integer | Нет | Связанная область | Должен существовать в таблице areas |
| tags | array | Нет | Список связанных тегов | Объекты с id, name, color |
| created_at | datetime | Нет (авто) | Время создания | ISO 8601 |
| updated_at | datetime | Нет (авто) | Время последнего обновления | ISO 8601 |
| completed_at | datetime | Нет (авто) | Время завершения | Устанавливается когда completed=true |
| deleted_at | datetime | Нет (авто) | Время мягкого удаления | Устанавливается при удалении |

**Связи**:
- Принадлежит: Project (необязательно)
- Принадлежит: Context (необязательно)
- Принадлежит: Area (необязательно)
- Имеет много: Tags (многие-ко-многим)

**Бизнес-правила для этой сущности**:
- Заголовок не может быть пустым
- Статус может переходить: inbox → active, inbox → someday, active → waiting, active → completed, waiting → active, someday → active, completed → active
- is_next_action может быть true только для задач со статусом в {active, inbox}
- Когда is_next_action устанавливается в true для задачи со статусом в {waiting, someday, completed}, статус должен измениться на active (с подтверждением пользователя)
- Когда completed устанавливается в true, статус изменяется на completed, waiting_for очищается, someday устанавливается в false
- deleted_at устанавливается вместо окончательного удаления (мягкое удаление)

### Сущность: Проект (Project)

| Поле | Тип | Обязательно | Описание | Ограничения |
|-------|------|------------|-------------|-------------|
| id | integer | Нет (авто) | Уникальный идентификатор | Автогенерируемый |
| name | string | Да | Название проекта | 1-100 символов |
| description | string | Нет | Детали проекта | Максимум 500 символов |
| area_id | integer | Нет | Связанная область | Должен существовать в таблице areas |
| status | enum | Нет (авто) | Статус проекта | active, completed (по умолчанию: active) |
| progress | integer | Нет (авто) | Процент завершения | 0-100 (вычисляемый) |
| total_tasks | integer | Нет (авто) | Общее количество задач | Вычисляется из tasks |
| completed_tasks | integer | Нет (авто) | Количество завершённых задач | Вычисляется из tasks |
| created_at | datetime | Нет (авто) | Время создания | ISO 8601 |
| updated_at | datetime | Нет (авто) | Время последнего обновления | ISO 8601 |

**Связи**:
- Принадлежит: Area (необязательно)
- Имеет много: Tasks

**Бизнес-правила для этой сущности**:
- Название не может быть пустым
- Progress = (completed_tasks / total_tasks) * 100 когда total_tasks > 0, иначе 0
- Когда статус изменяется на completed, все связанные задачи могут быть завершены (необязательное поведение)
- Удаление проекта не каскадно удаляет задачи

**Бизнес-правила для этой сущности**:

## 4. Основная бизнес-логика

### Логика: Вычисление прогресса задач для проектов
- **Назначение**: Автоматическое вычисление процента завершения проекта на основе связанных задач
- **Триггер**: Когда задачи получаются для проекта или списка проектов
- **Алгоритм**:
  1. Получить все задачи, связанные с проектом (фильтр по project_id)
  2. Посчитать общее количество задач: total_tasks = tasks.length
  3. Посчитать завершённые задачи: completed_tasks = tasks.filter(t => t.completed).length
  4. Вычислить прогресс: 
     - Если total_tasks > 0: progress = Math.round((completed_tasks / total_tasks) * 100)
     - Иначе: progress = 0
- **Краевые случаи**:
  - Проект без задач: progress = 0, total_tasks = 0, completed_tasks = 0
- **Заметки по производительности**: Вычисление происходит на стороне клиента после получения данных из API

### Логика: Управление статусом следующего действия
- **Назначение**: Обеспечить соблюдение бизнес-правил при отмечении задач как следующие действия
- **Триггер**: Пользователь кликает переключатель "Next Action" на задаче
- **Алгоритм**:
  1. Проверить текущий статус задачи и значение is_next_action
  2. Если устанавливается в true (отмечаем как следующее действие):
     - Если статус задачи в {waiting, someday, completed}:
       - Показать диалог подтверждения: "Задача в статусе [Статус]. Установка как следующего действия изменит статус на Активный. Продолжить?"
       - Если пользователь подтверждает: Обновить статус на "active", затем установить is_next_action = true
       - Если пользователь отменяет: Ничего не делать
     - Иначе (статус inbox или active): Установить is_next_action = true
  3. Если устанавливается в false: Установить is_next_action = false
- **Краевые случаи**:
  - Задача завершена и отмечена как следующее действие: требует изменения статуса на active
- **Заметки по производительности**: Мгновенное обновление UI с оптимистичным обновлением, подтверждается ответом API

### Логика: Поток обработки Inbox
- **Назначение**: Направлять пользователей через эффективную обработку элементов Inbox
- **Триггер**: Пользователь кликает кнопку "Process All"
- **Алгоритм**:
  1. Получить список активных задач Inbox (фильтр: status='inbox', completed=false)
  2. Если нет задач: Показать сообщение "Нет элементов для обработки"
  3. Установить флаг режима обработки = true
  4. Открыть форму редактирования для первой задачи в списке
  5. Когда пользователь сохраняет задачу:
     - Обновить задачу с предоставленными данными (должно включать изменение статуса на active)
     - Найти индекс текущей задачи в списке
     - Если следующая задача существует: Открыть форму для следующей задачи
     - Иначе (нет больше задач): Выйти из режима обработки, закрыть форму
  6. Когда пользователь удаляет задачу:
     - Удалить задачу из системы
     - Найти индекс текущей задачи в списке
     - Если следующая задача существует: Открыть форму для следующей задачи
     - Иначе (нет больше задач): Выйти из режима обработки, закрыть форму
  7. Когда пользователь отменяет или закрывает форму: Выйти из режима обработки
- **Краевые случаи**:
  - Пользователь редактирует задачу, но не меняет статус с inbox: задача остаётся в inbox
  - Задача удалена во время обработки: автоматически переходит к следующей
- **Заметки по производительности**: Оптимистичные обновления для мгновенной обратной связи

### Логика: Оптимистичные обновления для операций с задачами
- **Назначение**: Обеспечить мгновенную обратную связь UI пока запрос API в процессе
- **Триггер**: Любая мутация задачи (создание, обновление, удаление, завершение, установка следующего действия, установка ожидания)
- **Алгоритм**:
  1. Сохранить текущее состояние (предыдущие данные) перед мутацией
  2. Отменить ожидающие запросы для затронутых кэшей
  3. Обновить локальный кэш с оптимистичным изменением немедленно
  4. Отправить запрос API
  5. При успехе:
     - Инвалидировать соответствующие запросы для получения свежих данных
     - Новые данные из API заменяют оптимистичные данные
  6. При ошибке:
     - Восстановить предыдущее состояние из шага 1
     - Показать сообщение об ошибке пользователю
  7. При завершении (независимо от успеха/ошибки):
     - Инвалидировать запросы для обеспечения согласованности кэша
- **Краевые случаи**:
  - Несколько быстрых мутаций: Каждая мутация хранит своё собственное предыдущее состояние
  - Тайм-аут сети: Откат к предыдущему состоянию
- **Заметки по производительности**: UI ощущается мгновенным, снижает воспринимаемую задержку

### Логика: Удаление задач (мягкое удаление)
- **Назначение**: Разрешить восстановление задач вместо окончательного удаления
- **Триггер**: Пользователь удаляет задачу
- **Алгоритм**:
  1. Отправить запрос удаления в API (устанавливает временную метку deleted_at)
  2. При успехе:
     - Удалить задачу из всех активных списков (inbox, tasks, next actions и т.д.)
     - Задача остаётся видимой в представлении корзины
     - Задачу можно восстановить (очищает deleted_at)
     - Задачу можно окончательно удалить (удаляет из базы данных)
  3. При ошибке: Показать ошибку, оставить задачу в списках
- **Краевые случаи**:
  - Задача в корзине когда связанный проект удаляется: Задача остаётся в корзине
- **Заметки по производительности**: Мягкое удаление - немедленный вызов API, подтверждение не требуется для удаления, но требуется для окончательного удаления

### Логика: Управление состоянием еженедельного обзора
- **Назначение**: Отслеживать, на каком шаге обзора находится пользователь во время еженедельного обзора
- **Триггер**: Пользователь переходит между шагами обзора
- **Алгоритм**:
  1. Поддерживать состояние текущего шага (один из: inbox, projects, next-actions, someday)
  2. Отображать соответствующий контент для текущего шага
  3. Каждый шаг независим, но концептуально последовательный
  4. Пользователь может свободно перемещаться между шагами
- **Краевые случаи**:
  - Пользователь завершает один шаг, но не другие: Состояние сохраняется за сессию
- **Заметки по производительности**: Состояние только на стороне клиента, не сохраняется

## 5. Потоки пользователей

### Поток: Быстрое создание задачи

```
[Дашборд или любая страница] → [Пользователь вводит заголовок задачи] → [Нажать Enter/Отправить] 
                                                                  ↓
                                                        [Задача создана в Inbox]
                                                        [Показать уведомление об успехе]
```

**Шаги**:
1. **Форма быстрого создания**: Пользователь видит поле ввода на дашборде или плавающую кнопку действия
2. **Ввести заголовок**: Пользователь вводит заголовок задачи (обязательно)
3. **Необязательное описание**: Пользователь может добавить описание (необязательно)
4. **Отправить**: Пользователь нажимает Enter или кликает кнопку Отправить
5. **Создать задачу**: Система создаёт задачу со status="inbox", priority="medium"
6. **Обратная связь**: Показать сообщение об успехе, задача появляется в Inbox

**Исход успеха**: Задача захвачена быстро без выхода из текущего контекста
**Исходы при ошибке**: Ошибка API - показать сообщение об ошибке, задача не создана

### Поток: Обработка Inbox

```
[Страница Inbox] → [Пользователь кликает "Process All"] → [Форма редактирования открывается для первой задачи] 
                                                                  ↓
                                                      [Пользователь заполняет детали]
                                                                  ↓
                                                        [Пользователь сохраняет задачу]
                                                                  ↓
                                           [Форма открывается для следующей задачи] ИЛИ [Обработка завершена]
```

**Шаги**:
1. **Перейти к Inbox**: Пользователь переходит на страницу /inbox
2. **Просмотреть элементы**: Система показывает список задач Inbox (status='inbox', completed=false)
3. **Process All**: Пользователь кликает кнопку "Process All"
4. **Первая задача**: Система открывает форму редактирования для первой задачи
5. **Уточнить задачу**: Пользователь заполняет проект, контекст, теги, приоритет, статус (обычно меняет на 'active')
6. **Сохранить**: Пользователь сохраняет задачу
7. **Следующая задача**: Система автоматически открывает форму для следующей задачи
8. **Повторять**: Шаги 5-7 повторяются для каждой задачи
9. **Завершить**: Когда нет больше задач, система показывает сообщение "Обработка завершена"

**Альтернативные пути**:
- **Удалить задачу**: Пользователь может удалить задачу из потока обработки, автоматически переходит к следующей
- **Отменить обработку**: Пользователь может отменить и выйти из режима обработки

**Исход успеха**: Все задачи Inbox уточнены и организованы
**Исходы при ошибке**: Ошибка API при сохранении - показать ошибку, разрешить повтор или пропустить задачу

### Поток: Установить задачу как следующее действие

```
[Задача в любом представлении] → [Пользователь кликает переключатель Next Action] 
                                                  ↓
                             [Задача в статусе waiting/someday/completed?]
                               ├─ Да → [Показать диалог подтверждения]
                               │         ├─ Подтвердить → [Изменить статус на active, установить is_next_action=true]
                               │         └─ Отменить → [Ничего не делать]
                               └─ Нет → [Установить is_next_action=true]
```

**Шаги**:
1. **Просмотреть задачу**: Пользователь видит задачу в списке (задачи, следующие действия и т.д.)
2. **Переключить следующее действие**: Пользователь кликает кнопку/переключатель Next Action
3. **Проверить статус**: Система проверяет текущий статус задачи
4. **Если проблемный статус** (waiting, someday, completed):
    - Показать подтверждение: "Задача в статусе [Статус]. Установка как следующего действия изменит статус на Активный. Продолжить?"
    - Если подтвердить: Обновить статус на "active", установить is_next_action=true
    - Если отменить: Ничего не делать
5. **Если хороший статус** (inbox, active):
    - Установить is_next_action=true

**Исход успеха**: Задача отмечена как следующее действие и появляется в представлении Следующие действия
**Исходы при ошибке**: Ошибка API - показать ошибку, откатить состояние переключателя

### Поток: Установить задачу как ожидание

```
[Задача в любом представлении] → [Пользователь кликает кнопку Waiting] → [Модальное окно ожидания открывается]
                                                                 ↓
                                               [Пользователь вводит кого/чего ожидаем]
                                                                 ↓
                                                     [Пользователь отправляет модальное окно]
                                                                 ↓
                                      [Статус задачи меняется на "waiting", waiting_for устанавливается]

**Шаги**:
1. **Просмотреть задачу**: Пользователь видит задачу в списке
2. **Кликнуть Waiting**: Пользователь кликает кнопку "Waiting" на задаче
3. **Открыть модальное окно**: Система открывает модальное окно с полем ввода
4. **Ввести детали**: Пользователь вводит кого или чего мы ожидаем (обязательно)
5. **Отправить**: Пользователь отправляет модальное окно
6. **Обновить задачу**: Система устанавливает status="waiting", waiting_for=значение, удаляет из следующих действий

**Исход успеха**: Задача в состоянии ожидания, отслеживается со значением waiting_for
**Исходы при ошибке**: Ошибка API - показать ошибку, оставить модальное окно открытым для повтора

### Поток: Еженедельный обзор

[Страница обзора] → [Выбрать шаг: Inbox] → [Обработать элементы Inbox]

5. **Выбор следующих действий**: Пользователь видит все активные задачи, может переключить is_next_action для следующей недели

7. **Свободная навигация**: Пользователь может свободно перемещаться между шагами в любом порядке

**Исход успеха**: Пользователь завершает еженедельный обзор, система организована на следующую неделю
**Исходы при ошибке**: Ошибки API во время обзора - показать ошибку, но разрешить продолжение

### Поток: Восстановить удалённую задачу

```
[Страница корзины] → [Пользователь видит удалённые задачи] → [Пользователь кликает Восстановить на задаче]

3. **Восстановить задачу**: Пользователь кликает кнопку "Восстановить" на задаче
4. **Очистить deleted_at**: Система очищает временную метку deleted_at
5. **Удалить из корзины**: Задача исчезает из представления корзины, появляется в соответствующих списках

**Исход успеха**: Задача восстановлена и доступна в обычных представлениях
**Исходы при ошибке**: Ошибка API - показать ошибку, задача остаётся в корзине

| Данные | Стратегия кэша | Заметки |
