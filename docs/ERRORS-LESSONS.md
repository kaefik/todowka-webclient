# Руководство по избеганию распространенных ошибок

Документ содержит анализ ошибок, допущенных при разработке todowka-webclient, и рекомендации по их предотвращению.

## Ошибки и уроки

### 1. Отсутствие метода в репозитории

**Ошибка:** `AttributeError: 'ProjectRepository' object has no attribute 'get_filtered'`

**Причина:** Метод `get_filtered` был реализован в `TaskRepository`, но отсутствовал в `ProjectRepository`. При попытке вызова этого метода в коде получали ошибку.

**Решение:** Добавил метод `get_filtered` в `ProjectRepository` с аналогичной функциональностью.

**Уроки:**
- ✅ При добавлении новых методов в базовые классы репозиториев обязательно реализуйте их во всех дочерних классах
- ✅ Используйте наследование и абстрактные базовые классы для обеспечения интерфейса
- ✅ Перед использованием метода проверяйте его наличие в классе (IDE подсказки)
- ✅ Тестируйте API endpoints сразу после добавления

---

### 2. Отсутствующий API endpoint

**Ошибка:** HTTP 422 при GET запросе к `/api/v1/tasks/next-actions`

**Причина:** Frontend вызывал несуществующий endpoint. Backend route был определен, но метод не был реализован в TaskService.

**Решение:** Добавил:
- GET endpoint в `app/routes/tasks.py`
- Метод `get_next_actions()` в `TaskService`

**Уроки:**
- ✅ При добавлении новой фичи в frontend сначала добавляйте соответствующие backend endpoints
- ✅ Проверяйте все API endpoints перед развертыванием (можно использовать curl или Postman)
- ✅ Следите за несоответствием между frontend API calls и backend routes
- ✅ Используйте автоматические тесты API для проверки всех endpoints

---

### 3. Несоответствие типов данных между backend и frontend

**Ошибка:**
- Frontend проверял `task.status === 'completed'`, но в Task не было поля `completed`
- Тип Task в frontend не соответствовал схеме TaskResponse в backend
- TypeScript выдавал ошибки компиляции

**Причина:** Типы данных не были синхронизированы между frontend и backend. Backend отправлял поле `completed: bool`, но frontend ожидал `status: 'completed'`.

**Решение:**
- Обновил тип `Task` в frontend, добавив поля: `completed`, `someday`, `tags`, `due_date`, `reminder_time`, `waiting_for`, `delegated_to`
- Изменил проверку в TaskItem на `task.completed`

**Уроки:**
- ✅ Синхронизируйте типы данных между backend и frontend
- ✅ При изменении backend схемы (Pydantic models) сразу обновляйте TypeScript типы
- ✅ Используйте автоматическую генерацию типов из OpenAPI спецификации (swagger)
- ✅ Запускайте `npm run typecheck` после изменений типов
- ✅ Следите за TypeScript ошибками в IDE

---

### 4. Неправильное использование полей сущностей

**Ошибка:** Фильтр `completed` не работал

**Причина:** Модель Task имеет два поля:
- `completed: bool` - boolean флаг выполнения
- `status: str` - строковый статус ('inbox', 'active', 'waiting', 'someday', 'delegated')

При отметке задачи выполненной менялось только поле `completed`, но фильтр искал по `status == 'completed'`, которого не существует.

**Решение:** В `TaskRepository.get_filtered` добавил проверку:
```python
if filters['status'] == 'completed':
    query = query.filter(Task.completed == True)
else:
    query = query.filter(Task.status == filters['status'])
```

**Уроки:**
- ✅ Ясно документируйте назначение каждого поля в модели данных
- ✅ При проектировании модели выбирайте между bool флагом и enum статусом, не смешивайте
- ✅ Для фильтрации используйте соответствующие поля сущности
- ✅ Тестируйте все фильтры после реализации
- ✅ Рассмотрите использование property-методов для сложных условий (например, `is_completed`)

---

### 5. Неверное оптимистичное обновление

**Ошибка:** При нажатии на галочку задача визуально не менялась

**Причина:** В `useCompleteTask` мутации оптимистичное обновление устанавливало:
```typescript
{ ...task, status: 'completed', ... }
```
Но в реальности менялось поле `completed: true`.

**Решение:** Исправил на:
```typescript
{ ...task, completed: true, updated_at: new Date().toISOString() }
```

**Уроки:**
- ✅ Оптимистичные обновления должны соответствовать реальным изменениям данных
- ✅ Проверяйте, что именно изменяет backend endpoint
- ✅ Используйте правильные поля при обновлении (bool для bool, string для string)
- ✅ Добавляйте `updated_at` при оптимистичном обновлении для корректной работы cache

### 6. Отсутствие функционала (кнопка без действия)

**Ошибка:** При нажатии на Edit ничего не происходило

**Причина:** В `TasksPage` не был передан обработчик `onEdit` в `TaskList`, не было состояния для хранения редактируемой задачи, не было модального окна.

**Решение:** Добавил:
- Состояние `selectedTask`
- Функции `handleEdit` и `handleSave`
- Компоненты `Modal` и `TaskForm`
- Загрузку данных для формы (projects, contexts, tags)

**Уроки:**
- ✅ При добавлении кнопки в UI обязательно реализуйте обработчик
- ✅ Следите за пропами компонентов (особенно optional)
- ✅ Реализуйте полный цикл: кнопка → обработчик → состояние → UI → сохранение
- ✅ Используйте существующие паттерны (как в InboxPage) для реализации аналогичного функционала
- ✅ Перед сдачей проверяйте каждый interactive элемент

## Общие рекомендации

### При разработке новых фичей:

1. **Backend-first подход:**
   - Сначала создайте/измените модели данных
   - Добавьте/измените repository методы
   - Добавьте/измените service методы
   - Добавьте/измените routes
   - Протестируйте API endpoints (curl, Postman, test script)

2. **Frontend-after-backend:**
   - Обновите типы данных согласно backend схемам
   - Добавьте/измените API client методы
   - Добавьте/измените React hooks
   - Добавьте/измените UI компоненты
   - Протестируйте функционал в браузере

3. **Типизация:**
   - Всегда запускайте `npm run typecheck` после изменений
   - Синхронизируйте типы между backend и frontend
   - Используйте TypeScript для отлавливания ошибок на этапе компиляции

4. **Тестирование:**
   - Тестируйте каждый API endpoint
   - Тестируйте каждый interactive элемент в UI
   - Проверяйте все фильтры и сортировки
   - Проверяйте оптимистичные обновления

5. **Код-ревью:**
   - Проверяйте наличие всех необходимых методов в классах
   - Проверяйте соответствие типов данных
   - Проверяйте правильное использование полей сущностей
   - Проверяйте, что все кнопки и ссылки работают

### Чек-лист перед коммитом:

- [ ] Все типы TypeScript проходят проверку (`npm run typecheck`)
- [ ] Нет ESLint ошибок (`npm run lint`)
- [ ] Все API endpoints протестированы
- [ ] Все кнопки в UI работают
- [ ] Все фильтры работают
- [ ] Оптимистичные обновления соответствуют реальным изменениям
- [ ] Типы frontend синхронизированы с backend схемами
- [ ] Нет отсутствующих методов в классах/репозиториях

## Памятка по типам данных

### Backend (Pydantic):
```python
class TaskResponse(BaseModel):
    completed: bool              # Boolean флаг
    status: str                 # Строка: 'inbox', 'active', 'waiting', ...
    # ... другие поля
```

### Frontend (TypeScript):
```typescript
interface Task {
  completed: boolean;           // Boolean флаг (не 'completed' строка!)
  status: TaskStatus;          // Строка: 'inbox', 'active', 'waiting', ...
  // ... другие поля
}
```

### При фильтрации:
- Используйте `completed: true/false` для поиска выполненных задач
- Используйте `status: 'inbox'/'active'/...` для поиска по статусу
- НЕ ИСПОЛЬЗУЙТЕ `status: 'completed'` - такого значения нет!
